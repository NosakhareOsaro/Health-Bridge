# Information Governance

> **This is a portfolio/demonstration document.** HealthBridge processes only synthetic
> data generated by [Synthea](https://github.com/synthetichealth/synthea) -- no real
> patient, service-user, or NHS data is used, stored, or referenced anywhere in this
> project. The templates below are provided to demonstrate familiarity with NHS-style
> information governance practice (DPIA structure, access control matrices, data
> classification), **not** as a compliance artifact for a real deployment. Before any real
> patient data touched a system built from this codebase, a genuine DPIA would need to be
> completed by a qualified Data Protection Officer / Caldicott Guardian, and this document
> would need to be replaced, not reused.

## Data classification

| Data element (as used in this project) | Real-world classification if this were live | Status here |
|---|---|---|
| Patient demographics (name, DOB, address) | Special category / Confidential patient information | Synthetic (Synthea) |
| Condition/diagnosis codes | Special category health data | Synthetic (Synthea) |
| Observations/measurements | Special category health data | Synthetic (Synthea) |
| OAuth2 client secrets (`services/fhir-api`) | Confidential | Dev-only placeholder values, not for reuse |
| Analytics dashboards (aggregated) | Would typically be de-identified/aggregate | Computed from synthetic data only |

## Data Protection Impact Assessment (DPIA) template

A real deployment processing patient data would need a completed DPIA covering (at
minimum) the sections below. This is a **template with this project's answers filled in
for illustration** -- a real DPIA is a live risk-assessment document owned by the data
controller, not a static file in a code repository.

### 1. Description of processing

- **What data:** Patient demographics, encounters, conditions, observations (FHIR R4
  resources; OMOP CDM v5.4 once transformed).
- **Source:** In this project, 100% synthetic (Synthea). In a real deployment, this would
  be sourced from an EHR/EPR via FHIR API integration.
- **Purpose:** Demonstration of a FHIR + OMOP CDM clinical data platform architecture;
  population health analytics.
- **Retention:** No retention policy is implemented (development stack, ephemeral Docker
  volumes). A real deployment requires an explicit, documented retention schedule.

### 2. Necessity and proportionality

- Is the processing necessary for the stated purpose? For this project: yes, as a
  demonstration; the purpose is inherently non-clinical.
- For a real deployment: this section must justify *why* identifiable/special-category
  data is needed rather than aggregate/de-identified data for each specific use case
  (e.g. population health dashboards typically only need aggregate counts, not
  patient-level records with names).

### 3. Risks identified (illustrative, not exhaustive)

| Risk | Likelihood (if real data) | Impact | Mitigation in this project | Mitigation required for real deployment |
|---|---|---|---|---|
| Unauthorized API access to FHIR resources | N/A (synthetic data) | N/A | SMART on FHIR OAuth2 + scope enforcement (`services/fhir-api`) | Production-grade IdP, MFA, audit logging, penetration testing |
| Rate-limit bypass / DoS | N/A | N/A | Redis-backed per-identity rate limiting | Production WAF, infrastructure-level rate limiting |
| Re-identification from "aggregate" dashboard data | N/A (synthetic) | N/A | N/A | Small-cell suppression, k-anonymity thresholds on population health measures |
| Vocabulary mapping errors causing miscoded conditions | Low (synthetic) | Low (synthetic) | dbt schema tests, Great Expectations suite | Full Athena vocabulary + clinical coding QA review |
| Secrets committed to source control | Low | Medium | `.env.example` pattern, `.gitignore`, no real credentials in this repo | Secrets manager (e.g. Vault/AWS Secrets Manager), pre-commit secret scanning |

### 4. Consultation

A real DPIA requires consultation with a Data Protection Officer, Caldicott Guardian (for
NHS contexts), and typically the ICO for high-risk processing. Not applicable here since
no real data is processed.

## Access control matrix

Illustrative role-based access model for the SMART on FHIR gateway (`services/fhir-api`),
matching the demo client registry in `app/core/security.py`:

| Role / client | Scope granted | Resources | Interaction |
|---|---|---|---|
| `healthbridge-demo-app` (patient-facing SMART app) | `patient/Patient.read`, `patient/Observation.read`, `patient/Condition.read`, `patient/Encounter.read`, `launch/patient` | Patient, Observation, Condition, Encounter | Read only, scoped to the launched patient |
| `healthbridge-analytics-service` (backend service) | `system/*.read` | All resource types | Read only, system-wide (for population-level extraction, not point-of-care) |
| Unauthenticated caller | none | `/metadata`, `/healthz`, `/.well-known/smart-configuration` | Public FHIR capability discovery only, per FHIR spec |

In a real deployment, this table would be maintained by whoever owns the IdP/client
registry, reviewed on a fixed cadence, and every row would map to an actual named
individual/system with an audit trail -- not a static in-memory dict as it is in this
demo (see `docs/adr/0002-hapi-fhir-over-custom-build.md` for related scope discussion).

## What would need to change before this touched real patient data

1. Replace the in-memory OAuth2 client/token registries in `services/fhir-api` with a
   persistent, encrypted, audited store.
2. Complete a real DPIA with the organization's DPO/Caldicott Guardian.
3. Load the full OHDSI Athena vocabulary under an appropriate license/data use agreement.
4. Add audit logging for every FHIR read/write and every dashboard query touching
   patient-level data.
5. Add small-cell suppression / minimum-count thresholds to any aggregate dashboard measure
   that could otherwise re-identify individuals in small subpopulations.
6. Formal penetration testing and a security review of the SMART on FHIR implementation
   (see `services/fhir-api/README.md`'s "known simplifications" section for specific gaps,
   e.g. no PKCE support yet).
