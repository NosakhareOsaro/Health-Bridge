# Reproducing HealthBridge

This document describes, step by step, how to set up your environment and rebuild
this entire project from scratch: the FHIR R4 gateway, the Synthea-to-OMOP-CDM ETL
pipeline, and the analytics dashboards.

All data used throughout is synthetic, generated by [Synthea](https://github.com/synthetichealth/synthea).
No real patient data is used anywhere in this project.

---

## 1. Prerequisites

Install the following before starting:

| Tool | Version | Purpose |
|------|---------|---------|
| Docker & Docker Compose | recent | runs Postgres, HAPI FHIR, Redis, and the services |
| Python | 3.12+ | FHIR gateway, ETL scripts, analytics app |
| Java (JRE/JDK) | 17+ | required to run the Synthea generator |
| Git | any recent | version control |

On macOS, Java can be installed without admin rights via Homebrew:

```bash
brew install openjdk@21
```

Verify installs:

```bash
docker --version
docker compose version
python3 --version
java --version
git --version
```

## 2. Clone the repository

```bash
git clone <this-repo-url>
cd HealthBridge
```

---

## 3. Bring up the full platform with Docker Compose

```bash
docker compose up -d
```

This starts six containers:

- `fhir-db` — Postgres backing HAPI FHIR (host port `5433`)
- `hapi-fhir` — the HAPI FHIR R4 server (host port `8080`)
- `omop-db` — Postgres backing the OMOP CDM v5.4 schema (host port `5434`)
- `redis` — used for rate limiting and caching by the FHIR gateway (host port `6379`)
- `fhir-api` — the FastAPI SMART-on-FHIR gateway (host port `8000`)
- `analytics-dashboard` — the Streamlit analytics app (host port `8501`)

Wait 15-30 seconds on first run for HAPI FHIR to finish booting, then confirm each
service is reachable:

| Service | URL |
|---------|-----|
| FHIR gateway OpenAPI docs | http://localhost:8000/docs |
| Raw HAPI FHIR metadata | http://localhost:8080/fhir/metadata |
| Analytics dashboards | http://localhost:8501 |

At this point the analytics dashboards will load but show no data — the OMOP CDM
tables are still empty. Continue to the next section to populate them.

If you hit a port conflict on your machine (e.g. a local Redis already running on
6379), override the port in a local `docker-compose.override.yml` — do not edit
`docker-compose.yml` itself:

```yaml
services:
  redis:
    ports:
      - "16379:6379"
```

## 4. Run the ETL pipeline (Synthea to FHIR to OMOP CDM v5.4)

```bash
cd services/etl
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
```

### 4.1 Generate synthetic patients

```bash
./scripts/generate_synthea.sh 25 Massachusetts
```

This downloads the Synthea jar (first run only) and generates 25 synthetic patients
as FHIR R4 bundles under `synthea/output/fhir`. You can change the patient count and
state as needed.

### 4.2 Create the staging schema

```bash
export OMOP_DB_ASYNC_URL="postgresql+asyncpg://omop:omop@localhost:5434/omop"
alembic upgrade head
```

### 4.3 Load FHIR bundles into staging

```bash
python3 scripts/load_fhir_to_staging.py --input-dir synthea/output/fhir
```

This reads `OMOP_DB_ASYNC_URL` from the environment (or pass an override directly —
see `python3 scripts/load_fhir_to_staging.py --help`). It parses each bundle
(handling US Core race/ethnicity extensions and panel-observation splitting) and
loads patients, encounters, conditions, and observations into the staging schema.

### 4.4 Transform staging data into OMOP CDM v5.4 with dbt

```bash
cd dbt_omop
export DBT_PROFILES_DIR=.
export OMOP_DB_HOST=localhost OMOP_DB_PORT=5434 OMOP_DB_USER=omop OMOP_DB_PASSWORD=omop OMOP_DB_NAME=omop
dbt build
cd ..
```

(These are also the defaults baked into `profiles.yml`, so the exports above are
only needed if your setup differs from the defaults.)

This builds the seven core OMOP tables (`person`, `observation_period`,
`visit_occurrence`, `condition_occurrence`, `measurement`, `observation`, `death`)
in the `cdm` schema, including ICD-10/SNOMED/LOINC concept mapping against the
seeded vocabulary subset in `dbt_omop/seeds/`.

> To use the full OHDSI vocabulary instead of the small seeded subset, download the
> vocabulary files from [Athena](https://athena.ohdsi.org/) (free account required)
> and follow the loading instructions in `services/etl/README.md`.

### 4.5 Run data quality checks and generate the characterization report

```bash
export OMOP_DB_SYNC_URL="postgresql+psycopg2://omop:omop@localhost:5434/omop"
python3 scripts/run_data_quality_checks.py
python3 scripts/generate_achilles_report.py
```

The first script runs the Great Expectations suite against the populated CDM
tables. The second generates a standalone HTML characterization report (population
summary, age/sex pyramid, condition prevalence, length-of-stay, mortality).

## 5. Use the FHIR gateway

```bash
cd ../fhir-api
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
```

Run the test suite (mocked, no Docker required):

```bash
pytest -v
```

Or run the gateway directly against the live HAPI FHIR instance (already running via
Docker Compose from step 3):

```bash
uvicorn app.main:app --reload --port 8000
```

Visit `http://localhost:8000/docs` for interactive OpenAPI documentation, including
SMART on FHIR OAuth2 token issuance, resource search, and scope-enforced writes.

## 6. Use the analytics dashboards

```bash
cd ../analytics
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
```

Set the database connection (matches `.env.example` in this directory):

```bash
export OMOP_DB_URL=postgresql://omop:omop@localhost:5434/omop
```

Run the test suite:

```bash
pytest -v
```

`test_measures.py` and `test_icd10_to_snomed.py` are integration tests against the
live OMOP CDM Postgres populated in step 4 (they skip automatically if the database
isn't reachable). `test_forecasting.py` and `test_equity.py` run against generated
synthetic data and need no database.

Run the dashboard directly (already running via Docker Compose, but useful for
local development):

```bash
streamlit run app/Home.py
```

Pages:

1. **Home** — summary metrics (person count, visit count, condition count, etc.)
2. **Population Health** — age/sex pyramid, disease incidence, readmission rate,
   length-of-stay trends, computed directly from OMOP CDM tables
3. **A&E Forecasting** — a Prophet model trained on a synthetic daily A&E
   attendance series, showing actual vs. forecast with a confidence interval
4. **Health Equity** — a folium choropleth over a synthetic grid of service areas
   with fabricated deprivation, chronic-disease, and life-expectancy metrics

To regenerate the synthetic A&E series and retrain the forecasting model:

```bash
python3 forecasting/generate_ae_attendance_data.py
python3 forecasting/train_prophet_model.py --input forecasting/data/ae_attendances.csv --forecast-days 30 --output forecasting/data/ae_forecast.csv
```

To regenerate the synthetic health-equity regions:

```bash
python3 equity/generate_equity_data.py
```

To use the standalone ICD-10 to SNOMED mapping utility:

```bash
python3 mapping/icd10_to_snomed.py <ICD-10 code>
```

## 7. Full teardown and clean rebuild

To verify a completely clean rebuild (useful after making changes):

```bash
docker compose down -v
docker compose up -d
```

This removes all containers and volumes, then rebuilds everything from the images.
You will need to re-run the ETL steps in section 4 afterward, since `-v` also wipes
the OMOP CDM data.

## 8. Linting

Each service uses [ruff](https://docs.astral.sh/ruff/):

```bash
cd services/<service-name>
ruff check .
```

## 9. Continuous Integration

`.github/workflows/ci.yml` runs linting, the full test suite for each service, and a
Docker build for all images on every push. See that file for the exact job
definitions if you want to reproduce the CI checks locally.

## Known limitations of this reproduction

- Only a small representative ICD-10/SNOMED/LOINC vocabulary subset is seeded by
  default — swap in the full OHDSI Athena vocabulary for real coverage.
- The Achilles-style report is a custom lightweight HTML report, not the official
  OHDSI Achilles R tool.
- The A&E attendance series and health-equity regions/metrics are entirely
  synthetic and fabricated for demonstration purposes.
- Analytics dashboards use Streamlit rather than Power BI (see
  `docs/adr/0003-dashboard-tooling-choice.md` for the reasoning).
